# Evaluation of Multi-stage Builds and BuildKit Layer Caching in Docker 🐳✨

Репозиторий содержит исходный код автоматизированного тестового стенда и исследовательские материалы для научной статьи **«Исследование эффективности совместного применения многоэтапных сборок и механизмов кэширования слоев Docker для оптимизации локального развертывания веб-приложений»** (конференция «Студенческая весна – 2026» 🌸).

**Цель проекта** — количественная оценка влияния архитектуры `Multi-stage` сборок и алгоритмов кэширования движка `BuildKit` на временные метрики и дисковый след локального окружения разработчика при симуляции реального рабочего цикла.

## 📁 Структура проекта

Тестовый стенд представляет собой полноценный аналитический веб-сервис на базе FastAPI со слоистой архитектурой (база данных, валидация, тяжелые математические вычисления) для создания реалистичной нагрузки на сборщик.

```text
docker_benchmark_project/
├── app/
│   ├── __init__.py
│   ├── database.py       # Настройка SQLAlchemy и подключение к SQLite
│   ├── models.py         # SQLAlchemy-модели (таблица User)
│   ├── schemas.py        # Pydantic-схемы для валидации запросов
│   ├── services.py       # Тяжелая аналитика матриц (Pandas, NumPy, SciPy)
│   ├── tasks.py          # Имитация фоновых задач (BackgroundTasks)
│   ├── main.py           # Точка входа в FastAPI приложение
│   └── requirements.txt  # Список тяжеловесных реальных зависимостей
├── Dockerfile.single     # Классический одноэтапный Dockerfile (базовый рантайм)
├── Dockerfile.multistage # Оптимизированный многоэтапный Dockerfile (разделённый рантайм)
├── Makefile              # Сценарии автоматизации подготовки и запуска под Windows 🛠️
└── pipeline.py           # Автоматизированный скрипт проведения 8 итераций теста

```

## 🚀 Инструкция по запуску и автоматизации

Управление стендом полностью инкапсулировано в локальный файл `Makefile`, что исключает рутину с ручным вводом путей и команд в терминале Windows.

### 1. Подготовка виртуального окружения 📦

Команда создает изолированное окружение `.venv` на базе установленного в Windows интерпретатора Python 3.10 и разворачивает туда необходимые пакеты для сборки метрик и отрисовки графиков:

```bash
make venv

```

### 2. Запуск эксперимента 📊

Запуск полностью автоматического пайплайна, который поочередно выполнит все 8 шагов эксперимента, соберет системные метрики и сохранит итоговую аналитику:

```bash
make bench

```

### 3. Алгоритм работы пайплайна (8 последовательных итераций)

Скрипт программно симулирует реальные действия разработчика, проходя через два ключевых блока испытаний:

* **Блок 1: Single-stage (Одноэтапная архитектура образа)**
* **Итерация 1 (Cold Start):** Полная очистка кэша сборщика (`docker builder prune -a -f`), холодная сборка базового образа.
* **Итерация 2 (Code Mod):** Автоматическое добавление строки-комментария в конец `app/main.py`. Замер времени горячего старта при стабильном кэше пакетов.
* **Итерация 3 (Dep Mod):** Добавление новой тяжелой библиотеки в `app/requirements.txt`. Фиксация полной инвалидации кэша и повторной установки всего стека зависимостей.
* **Итерация 4 (No Change):** Повторная сборка без изменений для проверки работы идеального кэша слоев.


* **Блок 2: Multi-stage (Многоэтапная оптимизированная архитектура)**
* **Итерация 5 (Cold Start):** Принудительная очистка кэша, чистая сборка двухстадийного образа (стадии `builder` и `runner`).
* **Итерация 6 (Code Mod):** Изменение строки кода в `app/main.py`. Проверка изоляции кода от тяжелых слоев зависимостей с помощью BuildKit.
* **Итерация 7 (Dep Mod):** Добавление новой библиотеки в `app/requirements.txt`. Демонстрация того, как пересобирается исключительно изолированная builder-стадия.
* **Итерация 8 (No Change):** Финальный проверочный запуск для валидации стабильности построенного графа сборки.



> **Важно:** После завершения модификаций файлов скрипт автоматически откатывает изменения в исходное состояние, оставляя репозиторий абсолютно чистым!
